# Custom Position Categories Plan

## Summary

Implement custom categories as a user-owned category layer on top of Foliofox’s fixed system categories.

When a user chooses a custom category, Foliofox will:

- Show only the custom category name in the UI.
- Store `positions.category_id = "other"` in the background.
- Store `positions.user_category_id = <custom category id>`.
- Group user-facing analytics, tables, filters, and AI summaries by the custom category.
- Keep narrow system-only behavior, like cash scenario suggestions, based on system categories only.

Every phase below must stop after completion and wait for explicit user approval before continuing.

## Phase 1 — Database Contract

Precondition: the user creates an empty Supabase migration file. The agent edits only that existing file.

- Add `user_position_categories` with `id`, `user_id`, `position_type`, `name`, optional `description`, `display_order`, timestamps.
- Add `positions.user_category_id uuid null`.
- Add FK/indexes for category joins and lookups; use `ON DELETE SET NULL` for deleted custom categories.
- Add case-insensitive unique category names per `user_id + position_type`.
- Add RLS so users can only select/insert/update/delete their own custom categories.
- Tighten position insert/update RLS so a user can only assign a custom category they own.
- Do not run Supabase CLI or apply the migration. The user applies it locally and regenerates `types/database.types.ts`.

## Phase 2 — Server Data Model

Precondition: database types have been regenerated by the user.

- Extend position types with:
  - `user_category_id`
  - `user_category_name`
  - `display_category_id`
  - `display_category_name`
  - `is_custom_category`
- Update `fetchPositions()` and `fetchSinglePosition()` to left join custom categories in the same base query and avoid N+1 reads.
- Add server functions/actions for:
  - fetching combined category options
  - creating a custom category
  - resolving/reusing a custom category by name for imports
- Update create/update position actions:
  - if `user_category_id` is present, force `category_id = "other"`
  - if a system category is selected, clear `user_category_id`
  - never trust client-submitted `category_id` for custom-category selections

## Phase 3 — Core UX

- Replace the current category selector behavior with a combined combobox:
  - system categories first
  - existing custom categories second
  - `Create custom category...` as the final entry
- Selecting `Create custom category...` opens inline creation inside the selector; created categories are selected immediately.
- In add/edit asset forms, show one field: `Category`. Do not show the hidden Foliofox `other` category when a custom category is selected.
- Update visible category labels across asset header, asset table grouping, position selector grouping, public portfolio table, and related badges to use `display_category_name`.
- Existing positions without custom categories continue to display their system category unchanged.

## Phase 4 — Analytics And AI

- Update asset allocation to group by `display_category_id/display_category_name`, so custom categories appear as first-class donut chart slices.
- Update allocation drift and AI portfolio overview to use display categories for user-facing category names.
- Keep system-only logic unchanged where it intentionally depends on fixed categories, especially:
  - scenario initial cash suggestion uses `category_id === "cash"`
  - custom categories are not treated as cash/crypto/domain unless the user selected the system category instead
- Document this tradeoff in code comments near the affected system-only logic.

## Phase 5 — Import And Export

- Export both canonical and custom category fields:
  - keep `category_id`
  - add `user_category`
- Import behavior:
  - `category_id` remains canonical and maps only to Foliofox system categories
  - optional `user_category` creates/reuses a custom category and stores system `category_id = "other"`
  - do not auto-create custom categories from arbitrary broker category labels
- Update import review UI to use the combined category selector.
- AI extraction remains system-category based; users can change extracted rows to custom categories in the review step.

## Test Plan

- Add focused tests for position create/update category normalization.
- Add fetch/transform tests covering system category, custom category, and deleted custom category fallback.
- Add asset allocation tests proving custom categories produce separate chart groups.
- Add CSV import/export round-trip tests for `user_category`.
- Add selector/component tests for selecting existing custom categories and creating a new one.
- Verification only with allowed commands: `npm run format:check`, `npm run lint`, targeted `npm run test -- ...`, and `npm run type`. No Supabase CLI, Next dev/build commands, or database commands.

## Assumptions

- Custom category selection means the user intentionally opts out of Foliofox’s fixed taxonomy for that position.
- Custom categories are asset-focused in the UI for v1, but the schema includes `position_type` for future liabilities.
- V1 includes inline create and reuse of custom categories, but no separate category management screen for rename/delete/reorder.
- Deleting a custom category, if done later, leaves affected positions as system `other`.
